## 1. 문제 (Problem)

소규모 모임(5~20명) 주최자가 겪는 3가지 핵심 고통:

1. **도구 파편화**: 이벤트 안내(카카오톡), 참여 신청(구글폼), 공지(단톡방)를 각각 다른 도구로 관리해야 함
2. **참여자 관리 비효율**: 승인/거절을 수작업으로 처리하고 현황을 별도로 기록해야 함
3. **소통 누락**: 공지가 채팅 메시지에 묻혀 중요 정보를 놓치거나 재공유해야 하는 번거로움

**기존 대안:**

- 단톡방 (카카오톡 오픈채팅)
- 구글폼 + 스프레드시트 조합
- 네이버 밴드, 클럽하우스 등

---

## 2. 고객 세그먼트 (Customer Segments)

**주요 고객:**

- 소규모 정기 모임(5~20명)을 직접 운영하는 **모임 주최자**
  - 예: 수영 모임, 헬스 스터디, 독서 클럽, 친구 정기 모임, 사내 소모임
  - 특징: 비기술적 사용자, 모임 운영 경험 있음, 반복적 이벤트 관리 필요

**얼리 어답터:**

- 참여자 관리가 번거로워 자동화 도구를 찾고 있는 모임 주최자
- 승인형 참여 관리가 필요한 소규모 커뮤니티 운영자

**참여자 (보조 세그먼트):**

- 주최자로부터 초대 링크를 받은 일반 모임 참여자

---

## 3. 가치 제안 (Unique Value Proposition)

> **"초대 링크 하나로 모임의 생성부터 공지까지, 모든 운영을 한 곳에서"**

- 이벤트 생성 → 초대 코드 공유 → 참여 신청/승인 → 공지 발송을 단일 서비스에서 완결
- 별도 앱 설치 없이 웹 브라우저로 즉시 사용 가능
- 역할(주최자/승인 참여자/대기 참여자)에 따라 딱 필요한 기능만 노출되는 깔끔한 UX

---

## 4. 해결책 (Solution)

| 문제               | 해결책                                                                                   |
| ------------------ | ---------------------------------------------------------------------------------------- |
| 도구 파편화        | 이벤트 생성·초대·참여자 관리·공지를 단일 웹 서비스로 통합 (F001~F009)                    |
| 참여자 관리 비효율 | 자유참여(즉시 승인) / 승인형 선택 + 참여자 관리 페이지에서 원클릭 승인·거절 (F005, F006) |
| 소통 누락          | 승인된 멤버 전용 공지 탭 + 댓글 기능으로 이벤트 내 소통 집중화 (F007~F009)               |

**MVP 핵심 기능:** 이벤트 생성, 초대 코드, 참여 신청/승인, 공지, 댓글

## 5. 경쟁우위 (Unfair Advantage)

- **낮은 진입 장벽**: 초대 코드 URL만 공유하면 비로그인 상태에서도 이벤트 내용 미리보기 가능 — 가입 장벽 최소화
- **역할 기반 접근 제어**: RLS(Row Level Security) 기반으로 주최자·승인 참여자·대기 참여자 권한 분리, 데이터 보안 내재화
- **워크플로우 완결성**: 외부 도구 연계 없이 모임 운영 전 사이클(초대→신청→승인→공지→댓글)을 단일 서비스 내에서 완결
- **빠른 반복 개발**: Next.js 15 + Supabase 조합으로 기능 추가·변경이 용이한 기술 구조

## 6. 수익 흐름 (Revenue Stream)

> MVP 단계에서는 수익 모델 미적용 (검증 우선)

**향후 검토 모델:**

- **Freemium**: 무료(이벤트 3개/월) + 유료(무제한 + 고급 기능)
- **프리미엄 기능**: 실시간 알림, 정산/결제 연동, 카풀 매칭, 이벤트 공개 목록 노출
- **팀/단체 플랜**: 다수의 모임을 운영하는 커뮤니티·기업 대상 구독
